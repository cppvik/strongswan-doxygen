1 # The Versatile IKE Control Interface (VICI) protocol #
2 
3 The vici plugin implements the server side of an IPC protocol to configure,
4 monitor and control the IKE daemon charon. It uses request/response and event
5 messages to communicate over a reliable stream based transport.
6 
7 ## Transport protocol ##
8 
9 To provide the service, the plugin opens a listening socket using a reliable,
10 stream based transport. charon relies on the different stream service
11 abstractions provided by libstrongswan, such as TCP and UNIX sockets.
12 
13 A client connects to this service to access functionality. It may send an
14 arbitrary number of packets over the connection before closing it.
15 
16 To exchange data, the transport protocol is segmented into byte sequences.
17 Each byte sequence is prefixed by a 32-bit length header in network order,
18 followed by the data. The maximum segment length is currently limited to 512KB
19 of data, and the length field contains the length of the data only, not
20 including the length field itself.
21 
22 The order of byte sequences must be strict, byte sequences must arrive in the
23 same order as sent.
24 
25 ## Packet layer ##
26 
27 Within the byte sequences defined by the transport layer, both the client
28 and the server can exchange packets. The type of packet defines its structure
29 and purpose. The packet type is a 8-bit identifier, and is the first byte
30 in a transport layer byte sequence. The length of the packet is given by the
31 transport layer.
32 
33 While a packet type may define the format of the wrapped data freely, currently
34 all types either contain a name, a message or both. The following packet types
35 are currently defined:
36 
37 * _CMD_REQUEST = 0_: A named request message
38 * _CMD_RESPONSE = 1_: An unnamed response message for a request
39 * _CMD_UNKNOWN = 2_: An unnamed response if requested command is unknown
40 * _EVENT_REGISTER = 3_: A named event registration request
41 * _EVENT_UNREGISTER = 4_: A named event deregistration request
42 * _EVENT_CONFIRM = 5_: An unnamed response for successful event (de-)registration
43 * _EVENT_UNKNOWN = 6_: A unnamed response if event (de-)registration failed
44 * _EVENT = 7_: A named event message
45 
46 For packets having a named type, after the packet type an 8-bit length header
47 of the name follows, indicating the string length in bytes of the name tag, not
48 including the length field itself. The name is an ASCII string that is not
49 null-terminated.
50 
51 The rest of the packet forms the exchanged message, the length is determined
52 by the transport byte sequence length, subtracting the packet type and
53 the optional name tag in some messages.
54 
55 ### Commands ###
56 
57 Commands are currently always requested by the client. The server replies with
58 a response, or with a CMD_UNKNOWN failure message to let the client know
59 that it does not have a handler for such a command. There is no sequence number
60 to associate responses to requests, so only one command can be active at
61 a time on a single connection.
62 
63 ### Events ###
64 
65 To receive event messages, the client explicitly registers for events by name,
66 and also unregisters if it does not want to receive events of the named kind
67 anymore. The server confirms event registration using EVENT_CONFIRM, or
68 indicates that there is no such event source with EVENT_UNKNOWN.
69 
70 Events may get raised at any time while registered, even during an active
71 request command. This mechanism is used to feed continuous data during a request,
72 for example.
73 
74 ## Message format ##
75 
76 The defined packet types optionally wrap a message with additional data.
77 Messages are currently used in CMD_REQUEST/CMD_RESPONSE, and in EVENT packets.
78 A message uses a hierarchial tree of sections. Each section (or the implicit
79 root section) contains an arbitrary set of key/value pairs, lists and
80 sub-sections. The length of a message is not part of the message itself, but
81 the wrapping layer, usually calculated from the transport byte sequence length.
82 
83 The message encoding consists of a sequence of elements. Each element starts
84 with the element type, optionally followed by an element name and/or an element
85 value. Currently the following message element types are defined:
86 
87 * _SECTION_START = 1_: Begin a new section having a name
88 * _SECTION_END = 2_: End a previously started section
89 * _KEY_VALUE = 3_: Define a value for a named key in the current section
90 * _LIST_START = 4_: Begin a named list for list items
91 * _LIST_ITEM = 5_: Define an unnamed item value in the current list
92 * _LIST_END = 6_: End a previously started list
93 
94 Types are encoded as 8-bit values. Types having a name (SECTION_START,
95 KEY_VALUE and LIST_START) have an ASCII string following the type, which itself
96 uses an 8-bit length header. The string must not be null-terminated, the string
97 length does not include the length field itself.
98 
99 Types having a value (KEY_VALUE and LIST_ITEM) have a raw blob sequence,
100 prefixed with a 16-bit network order length. The blob follows the type or the
101 name tag if available, the length defined by the length field does not include
102 the length field itself.
103 
104 The interpretation of any value is not defined by the message format; it can
105 take arbitrary blobs. The application may specify types for specific keys, such
106 as strings or integer representations. The vici plugin currently uses
107 non-null terminated strings as values only; numbers get encoded as strings.
108 
109 ### Sections ###
110 
111 Sections may be opened in the implicit root section, or any previously section.
112 They can be nested to arbitrary levels. A SECTION_END marker always closes
113 the last opened section; SECTION_START and SECTION_END items must be balanced
114 in a valid message.
115 
116 ### Key/Values ###
117 
118 Key/Value pair elements may appear in the implicit root section or any explicit
119 sub-section at any level. Key names must be unique in the current section, use
120 lists to define multiple values for a key. Key/values may not appear in lists,
121 use a sub-section instead.
122 
123 ### Lists ###
124 
125 Lists may appear at the same locations as Key/Values, and may not be nested.
126 Only a single list may be opened at the same time, and all lists must be closed
127 in valid messages. After opening a list, only list items may appear before the
128 list closing element. Empty lists are allowed, list items may appear within
129 lists only.
130 
131 ### Encoding example ###
132 
133 Consider the following structure using pseudo-markup for this example:
134 
135  key1 = value1
136  section1 = {
137  sub-section = {
138  key2 = value2
139  }
140  list1 = [ item1, item2 ]
141  }
142 
143 The example above reprensents a valid tree structure, that gets encoded as
144 the following C array:
145 
146  char msg[] = {
147  /* key1 = value1 */
148  3, 4,'k','e','y','1', 0,6,'v','a','l','u','e','1',
149  /* section1 */
150  1, 8,'s','e','c','t','i','o','n','1',
151  /* sub-section */
152  1, 11,'s','u','b','-','s','e','c','t','i','o','n',
153  /* key2 = value2 */
154  3, 4,'k','e','y','2', 0,6,'v','a','l','u','e','2',
155  /* sub-section end */
156  2,
157  /* list1 */
158  4, 5, 'l','i','s','t','1',
159  /* item1 */
160  5, 0,5,'i','t','e','m','1',
161  /* item2 */
162  5, 0,5,'i','t','e','m','2',
163  /* list1 end */
164  6,
165  /* section1 end */
166  2,
167  };

169 ## Client-initiated commands ##
170 
171 Based on the packet layer, VICI implements commands requested by the client
172 and responded to by the server using named _CMD_REQUEST_ and _CMD_RESPONSE_
173 packets wrapping messages. The request message may contain command arguments,
174 the response message the reply.
175 
176 Some commands use response streaming, that is, a request triggers a series of
177 events to consecutively stream data to the client before the response message
178 completes the stream. A client must register for the appropriate event to
179 receive the stream, and unregister after the response has been received.
180 
181 The following client issued commands with the appropriate command input and
182 output messages are currently defined:

820 # libvici C client library #
821 
822 libvici is the reference implementation of a C client library implementing
823 the vici protocol. It builds upon libstrongswan, but provides a stable API
824 to implement client applications in the C programming language. libvici uses
825 the libstrongswan thread pool to deliver event messages asynchronously.
826 
827 ## Connecting to the daemon ##
828 
829 This example shows how to connect to the daemon using the default URI, and
830 then perform proper cleanup:
831 
832  #include <stdio.h>
833  #include <errno.h>
834  #include <string.h>
835 
836  #include <libvici.h>
837 
838  int main(int argc, char *argv[])
839  {
840  vici_conn_t *conn;
841  int ret = 0;
842 
843  vici_init();
844  conn = vici_connect(NULL);
845  if (conn)
846  {
847  /* do stuff */
848  vici_disconnect(conn);
849  }
850  else
851  {
852  ret = errno;
853  fprintf(stderr, "connecting failed: %s\n", strerror(errno));
854  }
855  vici_deinit();
856  return ret;
857  }

859 ## A simple client request ##
860 
861 In the following example, a simple _version_ request is issued to the daemon
862 and the result is printed:
863 
864  int get_version(vici_conn_t *conn)
865  {
866  vici_req_t *req;
867  vici_res_t *res;
868  int ret = 0;
869 
870  req = vici_begin("version");

872  if (res)
873  {
874  printf("%s %s (%s, %s, %s)\n",
875  vici_find_str(res, "", "daemon"),
876  vici_find_str(res, "", "version"),
877  vici_find_str(res, "", "sysname"),
878  vici_find_str(res, "", "release"),
879  vici_find_str(res, "", "machine"));
880  vici_free_res(res);
881  }
882  else
883  {
884  ret = errno;
885  fprintf(stderr, "version request failed: %s\n", strerror(errno));
886  }
887  return ret;
888  }

890 ## A request with event streaming and callback parsing ##
891 
892 In this more advanced example, the _list-conns_ command is used to stream
893 loaded connections with the _list-conn_ event. The event message is parsed
894 with a simple callback to print the connection name:
895 
896  int conn_cb(void *null, vici_res_t *res, char *name)
897  {
898  printf("%s\n", name);
899  return 0;
900  }
901 
902  void list_cb(void *null, char *name, vici_res_t *res)
903  {
904  if (vici_parse_cb(res, conn_cb, NULL, NULL, NULL) != 0)
905  {
906  fprintf(stderr, "parsing failed: %s\n", strerror(errno));
907  }
908  }
909 
910  int list_conns(vici_conn_t *conn)
911  {
912  vici_req_t *req;
913  vici_res_t *res;
914  int ret = 0;
915 
916  if (vici_register(conn, "list-conn", list_cb, NULL) == 0)
917  {
918  req = vici_begin("list-conns");
919  res = vici_submit(req, conn);
920  if (res)
921  {
922  vici_free_res(res);
923  }
924  else
925  {
926  ret = errno;
927  fprintf(stderr, "request failed: %s\n", strerror(errno));
928  }
929  vici_register(conn, "list-conn", NULL, NULL);
930  }
931  else
932  {
933  ret = errno;
934  fprintf(stderr, "registration failed: %s\n", strerror(errno));
935  }
936  return ret;
937  }

944 # vici ruby gem #
945 
946 The _vici ruby gem_ is a pure ruby implementation of the VICI protocol to
947 implement client applications. It is provided in the _ruby_ subdirectory, and
948 gets built and installed if strongSwan has been _./configure_'d with
949 _--enable-vici_ and _--enable-ruby-gems_.
950 
951 The _Connection_ class from the _Vici_ module provides the high level interface,
952 the underlying classes are usually not required to build ruby applications
953 using VICI. The _Connection_ class provides methods for the supported VICI
954 commands and an event listening mechanism.
955 
956 To represent the VICI message data tree, the gem converts the binary encoding
957 to ruby data types. The _Connection_ class takes and returns ruby objects for
958 the exchanged message data:
959  * Sections get encoded as Hash, containing other sections as Hash, or
960  * Key/Values, where the values are Strings as Hash values
961  * Lists get encoded as Arrays with String values
962 Non-String values that are not a Hash nor an Array get converted with .to_s
963 during encoding.
964 
965 ## Connecting to the daemon ##
966 
967 To create a connection to the daemon, a socket can be passed to the
968 _Connection_ constructor. If none is passed, a default Unix socket at
969 _/var/run/charon.vici_ is used:
970 
971  require "vici"
972  require "socket"
973 
974  v = Vici::Connection.new(UNIXSocket.new("/var/run/charon.vici"))
975 
976 ## A simple client request ##
977 
978 An example to print the daemon version information is as simple as:
979 
980  x = v.version
981  puts "%s %s (%s, %s, %s)" % [
982  x["daemon"], x["version"], x["sysname"], x["release"], x["machine"]
983  ]
984 
985 ## A request with closure invocation ##
986 
987 The _Connection_ class takes care of event streaming by invoking a closure
988 for each event. The following example lists all loaded connections using the
989 _list-conns_ command and implicitly the _list-conn_ event:
990 
991  v.list_conns { |conn|
992  conn.each { |key, value|
993  puts key
994  }
995  }

1002 # vici Python egg #
1003 
1004 The _vici Python egg_ is a pure Python implementation of the VICI protocol to
1005 implement client applications. It is provided in the _python_ subdirectory, and
1006 gets built and installed if strongSwan has been _./configure_'d with
1007 _--enable-vici_ and _--enable-python-eggs_.
1008 
1009 The _vici_ module provides a _Session()_ constructor for a high level interface,
1010 the underlying classes are usually not required to build Python applications
1011 using VICI. The _Session_ class provides methods for the supported VICI
1012 commands.
1013 
1014 To represent the VICI message data tree, the library converts the binary
1015 encoding to Python data types. The _Session_ class takes and returns Python
1016 objects for the exchanged message data:
1017  * Sections get encoded as OrderedDict, containing other sections, or
1018  * Key/Values, where the values are strings as dictionary values
1019  * Lists get encoded as Python Lists with string values
1020 Values that do not conform to Python dict or list get converted to strings using
1021 str().
1022 
1023 ## Connecting to the daemon ##
1024 
1025 To create a connection to the daemon, a socket can be passed to the _Session_
1026 constructor. If none is passed, a default Unix socket at _/var/run/charon.vici_
1027 is used:
1028 
1029  import vici
1030  import socket
1031 
1032  s = socket.socket(socket.AF_UNIX)
1033  s.connect("/var/run/charon.vici")
1034  v = vici.Session(s)
1035 
1036 ## A simple client request ##
1037 
1038 An example to print the daemon version information is as simple as:
1039 
1040  ver = v.version()
1041 
1042  print "{daemon} {version} ({sysname}, {release}, {machine})".format(**ver)
1043 
1044 ## A request with response iteration ##
1045 
1046 The _Session_ class returns an iterable Python generator for streamed events to
1047 continuously stream objects to the caller. The following example lists all
1048 loaded connections using the _list-conns_ command and implicitly the _list-conn_
1049 event:
1050 
1051  for conn in v.list_conns():
1052  for key in conn:
1053  print key
1054 
1055 Please note that if the returned generator is not iterated completely, it must
1056 be closed using _close()_. This is implicitly done when breaking from a loop,
1057 but an explicit call may be required when directly iterating the generator with
1058 _next()_.
1059 
1060 ## Sorting in dictionaries ##
1061 
1062 In VICI, in some message trees the order of objects in dictionary matters. In
1063 contrast to ruby Hashes, Python dictionaries do not preserve order of added
1064 objects. It is therefore recommended to use OrderedDicts instead of the default
1065 dictionaries. Objects returned by the library use OrderedDicts.
